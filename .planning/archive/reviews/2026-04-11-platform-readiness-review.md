# HNF1B-DB Platform Readiness Review

Date: 2026-04-11
Scope: user management, CRUD, internal data curation, identity lifecycle, audit/collaboration, ORCID and public micro-attribution, contributor analytics, API/session security.
Method: 3 parallel codebase reality-checks (backend auth, backend CRUD/audit, frontend/planning), 3 parallel web-research agents (auth/session 2025-2026, ORCID + biocuration attribution, collaborative curation UX), Playwright UI walk, 1 follow-up research agent on dev-mode quick-login patterns.

---

## 1. Executive summary

HNF1B-DB is **close to ready** as a small invited-curator data-entry tool and **not yet ready** as a multi-user curation platform with modern identity lifecycle, collaboration, opt-in micro-attribution, contributor analytics, and hardened API/session security.

The existing auth baseline is stronger than it first appears:

- JWT access + **rotating refresh tokens with reuse-detection cascade** (matches RFC 9700 §2.2.2/§4.14), admin-only user CRUD, RBAC, time-based account lockout, Redis-backed rate limiter, fail-fast secret validators, security-headers middleware, RFC 6902 JSON Patch diffs on every audit row, curator-gated phenopacket CRUD with optimistic locking.

But the platform gaps are wider than login itself:

- No invite / password-reset / email-verification lifecycle.
- No admin user-management UI (backend is done — frontend is missing).
- No internal comment threads, no review state machine, no revision-history UI.
- No ORCID linking, no opt-in public micro-attribution, no user preferences, no private contributor dashboard.
- Front-end tokens still in `localStorage` rather than HttpOnly cookies.
- A concrete subset of OWASP API Security Top 10 2023 risks (BOPLA mass assignment, BFLA router guards, sensitive-business-flow bulk export) is not systematically defended.

**Bottom line:** the shortest credible path is to ship a small foundation bundle first (dev-mode quick-login + audit-on-create + DELETE revision check + FK-ify audit actor + global soft-delete filter), then an admin user-management UI as a thin wrapper on the existing backend — at which point the rest of the work (identity lifecycle, comments + review workflow, ORCID, preferences, session hardening) can land in independent bundles on a clean base.

Same issue on `updated_by`, `deleted_by`, and `PhenopacketAudit.changed_by`.
- `frontend/src/components/phenopacket/MetadataCard.vue:133-135, 195-207` renders `metaData.updates[*].reviewer`, but **no code path ever sets that field** — not the backend, not the frontend. Silent data gap. Confirmed visually in §4: live data shows `"Created by hnf1b-db v1.0.0"` (migration string), never a real reviewer.
- `backend/app/auth/user_import_service.py` is a **dead utility** — well-structured batch creator and secure password generator, but no endpoint, no migration, and no caller invokes it at runtime. Migration pipeline uses `reviewer_mapper.py` directly instead.

**Audit holes**

- **Audit entries are NOT auto-created on CREATE.** `create_phenopacket()` at `backend/app/phenopackets/routers/crud.py:312-363` never calls `create_audit_entry()` — only UPDATE and DELETE log audit rows. Small fix to land *before* any revision-history UI.
- **Optimistic locking is missing on DELETE.** `backend/app/phenopackets/routers/crud.py:468-542` doesn't check `revision`, so a stale client can race a concurrent edit and delete the newer copy.
- **No global soft-delete filter.** Every query must manually add `.where(Phenopacket.deleted_at.is_(None))`. Add one router without that guard and deleted records leak.

**ORCID, preferences, contributor stats**

- `User` model has no ORCID fields, no authentication linkage, no attribution-consent flag, no public-profile preference.
- No ORCID sign-in / link button anywhere in the frontend.
- No distinction between internal actor identity and public attribution identity.
- No user-preferences page — `/user` view shows read-only personal info + "Change Password" only. No ORCID toggle, no attribution toggle, no display-name preference.
- No private contributor dashboard — no "my created / reviewed / approved / unresolved / recent activity" surface.

**Security posture gaps**

- Password hashing uses **bcrypt via passlib** at `backend/app/auth/password.py:8`. OWASP's current first choice is Argon2id; more importantly, **`passlib` will not work on Python 3.13+** (see `pwdlib` README), so any "add argon2 to passlib CryptContext" plan is a dead end.
- Refresh tokens stored **raw** in `users.refresh_token` (no hashing). The rotation + reuse-detection logic is strong, but the raw storage means a DB snapshot = account takeover on every live session.
- Front-end tokens in **`localStorage`** rather than HttpOnly cookies. Axios interceptor reads them synchronously on every request.
- Route guards check `meta.requiresAdmin` / `authStore.accessToken` but don't re-validate with the backend — a mid-session role change takes effect only on the next API call that hits a `require_admin` dependency.
- No step-up reauth for sensitive actions, no session inventory, no forced logout, no documented API security baseline aligned to OWASP API Security Top 10 2023.
- `frontend/src/views/About.vue:50,79,131` and `frontend/src/views/FAQ.vue:56,64,73,139,177` use `v-html="renderMarkdown(...)"` — `frontend/src/utils/sanitize.js` (DOMPurify) exists but those paths need a verified wrapping. ESLint flagged 21 `v-html` warnings in the 2026-04-09 codebase-quality review.

## 3. Best practices research — 2025-2026 deltas for this stack

The base OWASP cheatsheets are still the right foundation. These are the concrete deltas that apply to HNF1B-DB's FastAPI + Vue stack:

### 3.1 Password hashing

- **`passlib` is effectively end-of-life and does not work on Python 3.13+** (see [`pwdlib` README](https://github.com/frankie567/pwdlib)). Any plan to "configure passlib's CryptContext for argon2" is a dead end.
- Use [**pwdlib**](https://github.com/frankie567/pwdlib) (written by the fastapi-users maintainer as an explicit passlib replacement): `PasswordHash.recommended()` returns Argon2id + a bcrypt verifier fallback, enabling **verify-and-rehash on next login** with zero forced logouts. Current [OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html) parameters for Argon2id: `m=19456, t=2, p=1`.

### 3.2 Browser token storage

- **[RFC 9700 (Jan 2025)](https://datatracker.ietf.org/doc/rfc9700/)** is now the authoritative OAuth 2.0 Security BCP and **mandates** refresh-token rotation *or* sender-constraint for public clients. HNF1B-DB's existing rotation implementation already satisfies this — the hardening work is storage + transport, not the rotation itself.
- [**draft-ietf-oauth-browser-based-apps §6.2**](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-browser-based-apps) recommends the hybrid pattern for sensitive-data SPAs: **access token in JS memory only, refresh token in an `HttpOnly + Secure + SameSite=Lax` cookie scoped to `/api/v2/auth/refresh`**. For HNF1B-DB (single-origin, clinical data) this is the right target — not a full BFF.
- Pair with the **signed double-submit CSRF pattern** from the [OWASP CSRF Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html): server sets a second non-HttpOnly `csrf` cookie, Axios interceptor copies it to an `X-CSRF-Token` header on state-changing requests. One focused migration PR; the Bearer-header path for the access token keeps working unchanged.

### 3.3 Registration / reset / invite flows

- **Token format**: **do not use JWT** for reset/invite/verify tokens — OWASP explicitly warns JWT can't be individually revoked, which breaks the single-use requirement. Use a server-side random 32-byte token, store a SHA-256 hash in a `credential_tokens(user_id, purpose, token_sha256, expires_at, used_at)` table — one table for reset + invite + verify.
- **Invite-only is the right default** for a curator cohort. Self-registration adds no value. The key footgun (from Auth0 community post-mortems): **bind the invite token to the target email at creation time** so a leaked invite can't be redirected.
- **Rate limits** (community norms from Auth0/Okta, OWASP doesn't publish numeric thresholds): login 5 / 15 min / account, reset-request 3 / h / account AND 10 / h / IP, registration/invite-accept 5 / h / IP, refresh 60 / min / session.

### 3.4 Session inventory & step-up auth

- A `sessions(id, user_id, device_fingerprint, user_agent, ip, created_at, last_seen_at, revoked_at)` table is the point at which pure-JWT flips to hybrid. Refresh-token rotation state gets bound to a `session_id`; deleting the row instantly revokes the whole family. This is the only honest way to implement "my active sessions" and "force logout on role change."
- Baseline for a curation platform: **15 min idle / 8 h absolute** timeouts (matches current [OWASP Session Management Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html) guidance for localStorage-stored JWTs).
- Step-up reauth triggers: password change, email change, MFA disable, role change (admin grant), ORCID link.

### 3.5 MFA / passkeys

- Passkeys are mainstream as of 2025 (Microsoft made passkeys default for new accounts May 2025; W3C WebAuthn L3 Working Draft Jan 2025). **Plan for passkeys; ship TOTP first** as the interim.
- Libraries: [`pyotp`](https://github.com/pyauth/pyotp) for TOTP; [`py_webauthn`](https://github.com/duo-labs/py_webauthn) (Duo Labs) for passkeys. Recovery codes: 10 single-use, stored as Argon2id hashes.

### 3.6 OWASP API Security Top 10 2023 — gaps most often missed

The two categories most often skipped on bespoke audits that will bite a curation platform with admin user CRUD:

- **API3 BOPLA (mass assignment).** Any `model.update(**request.dict())` in admin user CRUD is a direct `role=admin` promotion vector. Use explicit `UserUpdatePublic` Pydantic schemas that exclude `role`, `is_active`, `is_superuser`, `refresh_token`. Never spread a request body into a SQLAlchemy model.
- **API6 Unrestricted Access to Sensitive Business Flows.** Cursor pagination + unauthenticated list endpoints = scrape-the-whole-dataset risk. Enforce per-user export quotas and a separate "bulk export" ACL distinct from "browse."

Full HNF1B-DB-specific mapping of all 10 risks is in the [OWASP API Security Top 10 2023](https://owasp.org/API-Security/editions/2023/en/0x11-t10/); the headline items for this codebase are API1 BOLA (draft access control at query time), API2 broken auth (hash the stored refresh token), API3 BOPLA, API5 BFLA (router-level `dependencies=[require_role("admin")]` on `/admin/*`), and API6 sensitive business flows.

### 3.7 ORCID integration

- **Public API tier is sufficient.** Free, registration-only, supports `/authenticate` + `openid` scopes. 12 req/s and 100k reads/day per client. No Member dues required unless you ever want to *write* to an ORCID record.
- **Use OIDC, not bare OAuth2.** ORCID supports the Basic OIDC profile — request `openid /authenticate` and you get a signed `id_token`. Use [**Authlib**](https://docs.authlib.org/en/latest/client/fastapi.html)'s generic OIDC client (no prebuilt ORCID client in fastapi-users).
- **Phase A (link) before Phase B (sign in).** Link-only keeps authorization and account-lifecycle decisions local; ORCID never implies an elevated role.
- **User-model storage fields** (all nullable, all opt-in):
  - `orcid_id VARCHAR(19) UNIQUE NULL` (format `0000-0000-0000-000X`; last char may be `X` per ISO 7064 Mod 11,2 checksum)
  - `orcid_authenticated_at TIMESTAMPTZ NULL`
  - `orcid_source ENUM('oauth_authenticated','manual_unverified','admin_curated')`
  - `orcid_display_name VARCHAR NULL` — snapshot of the ORCID record's `name` at link time, kept separate from local `full_name` so internal audit identity ≠ public attribution identity
  - `orcid_public_opt_in BOOLEAN DEFAULT FALSE`
  - `orcid_public_opt_in_at TIMESTAMPTZ NULL` — audit trail for GDPR consent
- **Two footguns to definitely not hit**:
  1. **Never display an ORCID iD publicly without the researcher's own opt-in.** `manual_unverified` and `admin_curated` must never be rendered as authenticated.
  2. **Never use email as the linking key.** ORCID iD is the only stable identity; emails in ORCID records are independently mutable. Enforce `UNIQUE` on `orcid_id`; on collision, refuse auto-merge and surface an admin-resolved conflict.
- **Prior art worth copying**:
  - [**UniProt community bibliography**](https://journals.plos.org/plosbiology/article?id=10.1371/journal.pbio.3001464) — ORCID login = citation credit.
  - [**APICURON**](https://academic.oup.com/database/article/doi/10.1093/database/baab019/6244733) — cross-database ORCID-keyed curation events with badges; a low-cost alternative to building a bespoke credit system.
  - **OBO ontologies** — per-term `dc:contributor` + ORCID URIs on every assertion.
  - **LOVD** — per-gene curator display (individual credit, no ORCID yet).
- **Credit frameworks**: [**CRediT (NISO Z39.104-2022)**](https://credit.niso.org/) has an explicit "Data curation" role (CC-BY licensed); tag each review event with a CRediT role for future citability. Nanopublications and DataCite-ORCID are interesting but deferred.

### 3.8 Collaborative curation UX

- **5+1 state machine**, not 7: `draft → in_review → changes_requested → approved → published` (+ `archived`). **Skip `superseded`** until phenopackets are actually versioned — YAGNI. Prior art: ClinGen VCI's `In-progress → Provisional → Approved → Published` ([PMC8764818](https://pmc.ncbi.nlm.nih.gov/articles/PMC8764818/)), MediaWiki FlaggedRevs, OpenReview, Reactome.
- **State lives on both the record and the revision.** Current state on `phenopackets.state` (indexable); immutable `phenopacket_revisions` rows per transition with `actor_id, from_state, to_state, reason`. Matches ClinGen VCI's snapshot model.
- **One generic `comments` table**, not per-record-type tables: `comments(id, record_type, record_id, parent_id, thread_root_id, author_id, body_markdown, body_rendered, anchor jsonb, resolved_at, resolved_by, created_at, updated_at, deleted_at)` + immutable `comment_edits(comment_id, editor_id, prev_body, edited_at)` log + thin `comment_mentions(comment_id, user_id)` join table. Matches the GitHub PR review comments data model. **Edit history MUST be immutable** — curators cannot be allowed to silently rewrite an assertion someone else already acted on.
- **Diff strategy for JSONB phenopackets**: server-side RFC 6902 JSON Patch (Python `jsonpatch`, **same library the audit table already uses**), rendered field-grouped in the UI with a human label map (`/subject/sex` → "Sex", `/phenotypic_features/*/type/id` → "HPO term"). Resolve HPO / MONDO IDs to labels on render. Two-pane raw-JSON fallback for curators who want it. Accessibility: never encode +/− by color alone (WCAG 1.4.1) — pair with leading `+`/`−`/`~` glyph and `aria-label`.
- **Single-screen editorial layout**: copy the GitHub PR 3-lane anatomy — top header (state badge + action buttons), left lane (60%, record form), right lane (40%, tabs: Conversation / Changes / History), bottom docked markdown composer with `@mention` autocomplete, soft gate on approval when unresolved comments exist.
- **Vue 3 building blocks that are ready now**: [Tiptap v3](https://tiptap.dev/docs/editor/getting-started/install/vue3) for markdown + mentions (MIT, first-class Vue 3), [json-diff-kit + json-diff-kit-vue](https://github.com/RexSkz/json-diff-kit) for the diff viewer, Vuetify's built-in `v-timeline` for activity feeds, [vuetify-pro-tiptap](https://github.com/yikoyu/vuetify-pro-tiptap) for a pre-skinned editor.

## 5. Dev-mode quick-login — design

Developer-experience feature: on the login page in dev only, show a row of buttons like "Log in as admin / curator / viewer" that one-click authenticate pre-seeded fixture users. Must be structurally impossible to ship to production.

### 5.1 Prior art — nobody mainstream ships this publicly

| Platform | Pattern | Gating | Takeaway |
|---|---|---|---|
| **Supabase Studio** | Impersonate-any-user dropdown; mints JWT with project service-role key | Studio-only; relies on service-role key never shipping to clients | Closest analog; still server-side |
| **django-hijack / django-loginas** | Admin list has "login as user X" button | Per-admin permission, not env flag | Production-grade impersonation, not a dev-quick-login |
| **fastapi-users** | No built-in dev login; recommended pattern is `dependency_overrides[current_user]` in tests only | Test-only; never reachable in a running app | Confirms nobody upstream ships a public "dev login" button |
| **Phoenix LiveDashboard** | Mounted only when explicitly added to router; wrapped in `if Mix.env() == :dev` | Compile-time `Mix.env()` check; the module doesn't exist in prod release | The cleanest "structurally cannot ship" example — the model to aim for |
| **Next.js `x-middleware-subrequest` CVE** | Undocumented header was the *only* gate keeping Middleware from being skipped | Forgeable from outside; no env gate | [Vercel postmortem](https://vercel.com/blog/postmortem-on-next-js-middleware-bypass): "do not let [one mechanism] be the sole method of protecting routes" |

**Nobody ships a public "Login as admin" button.** The closest patterns are server-side admin impersonation guarded by service-role keys, or framework conventions that compile the feature out entirely. We want **both halves**: server-side auth that only exists at runtime in dev + compile-time elimination on the client.

### 5.2 Threat model & incident lessons

The failure mode is not "attacker steals a dev password." It's **"a dev-only feature shipped to production by accident"** — a flag left in an env file, a component import that wasn't tree-shaken, a route that wasn't removed. Every major recent incident in this family has the same shape: one flag carried the entire security weight. [CWE-489](https://cwe.mitre.org/data/definitions/489.html) catalogs the pattern.

| Incident | What happened | Prevention |
|---|---|---|
| **Django `DEBUG=True` epidemic** ([BleepingComputer](https://www.bleepingcomputer.com/news/security/misconfigured-django-apps-are-exposing-secret-api-keys-database-passwords/)) | 28k+ prod apps exposing debug pages with SECRET_KEY / DB creds; [escalation to RCE/SSRF](https://blog.vidocsecurity.com/blog/escalation-of-debug-mode-in-django) documented | Default to the locked-down value; refuse to start on unsafe combinations |
| **Apache Superset CVE-2023-27524** ([Horizon3](https://horizon3.ai/attack-research/disclosures/cve-2023-27524-insecure-default-configuration-in-apache-superset-leads-to-remote-code-execution/)) | Default `SECRET_KEY` shipped in install bundle; 2,124 / 3,176 internet-exposed instances used the default → trivial admin cookie forgery | **Refuse to start** when the secret matches a known-default — the model for our `model_validator` |
| **Next.js Middleware bypass** ([Vercel](https://vercel.com/blog/postmortem-on-next-js-middleware-bypass)) | One undocumented header was the only auth gate | Defense in depth — never let one toggle be load-bearing |
| **Spring Boot Actuator CVE-2026-22731/22733** ([HeroDevs](https://www.herodevs.com/blog-posts/spring-boot-authentication-bypass-two-new-cves-that-enterprise-teams-cannot-afford-to-ignore-cve-2026-22731-cve-2026-22733)) | Endpoints sharing a path prefix with debug routes inherited weakened auth | Physically separate router that isn't even imported in prod |
| **CWE-489 family** | "Special endpoints, test accounts, bypass mechanisms" left enabled in prod | Remove vs. disable — the safest debug code is debug code that doesn't exist in the binary |

Defense is layered so any single mistake fails closed.

### 5.3 Five-layer gating recipe

```
Layer 1 (backend config):   settings.environment + settings.enable_dev_auth
Layer 2 (backend router):   /api/v2/dev/* only mounted if both are true
Layer 3 (backend fixture):  dev user seeds with is_fixture_user=true flag
Layer 4 (frontend build):   import.meta.env.DEV gate on the component
Layer 5 (CI + ops):         grep-based PR check + prod compose asserts unset
```

#### Layer 1 — `backend/app/core/config.py`

```python
from typing import Literal
from pydantic import model_validator

class Settings(BaseSettings):
    # ... existing fields ...
    environment: Literal["development", "staging", "production"] = "production"
    enable_dev_auth: bool = False

    @model_validator(mode="after")
    def _refuse_dev_auth_in_prod(self) -> "Settings":
        if self.enable_dev_auth and self.environment != "development":
            raise ValueError(
                "REFUSING TO START: ENABLE_DEV_AUTH=true is only permitted "
                f"when ENVIRONMENT=development (got {self.environment!r})."
            )
        return self
```

`environment="production"` default is critical — **an unset env must never be interpreted as dev.** Same fail-fast pattern as the existing `JWT_SECRET` / `ADMIN_PASSWORD` validators at `backend/app/core/config.py:306-346`.

#### Layer 2 — `backend/app/api/dev_endpoints.py`

Belt-and-suspenders defenses: **module-level `assert`** (accidental import outside dev crashes on load), **`include_in_schema=False`** ([FastAPI conditional OpenAPI](https://fastapi.tiangolo.com/how-to/conditional-openapi/) — cosmetic, not security), and a **loopback-only `Depends`** guard.

```python
# backend/app/api/dev_endpoints.py
"""Dev-only quick-login. NEVER IMPORTED IN PRODUCTION."""
import logging
from fastapi import APIRouter, Depends, HTTPException, Request, status
from sqlalchemy.ext.asyncio import AsyncSession

from app.auth.tokens import create_access_token, create_refresh_token
from app.core.config import settings
from app.database import get_db
from app.repositories.user_repository import UserRepository
from app.schemas.auth import Token

logger = logging.getLogger(__name__)
router = APIRouter(prefix="/api/v2/dev", tags=["dev-only"], include_in_schema=False)

# Hard runtime assert — belt + suspenders for the import gate
assert settings.enable_dev_auth and settings.environment == "development", (
    "dev_endpoints.py was imported outside of dev mode — investigate main.py"
)

_LOOPBACK = {"127.0.0.1", "::1", "localhost", "testclient"}

def _require_loopback(request: Request) -> None:
    host = request.client.host if request.client else None
    if host not in _LOOPBACK:
        raise HTTPException(status.HTTP_403_FORBIDDEN, detail="loopback only")

@router.post("/login-as/{username}", response_model=Token)
async def dev_login_as(
    username: str,
    request: Request,
    db: AsyncSession = Depends(get_db),
    _: None = Depends(_require_loopback),
) -> Token:
    repo = UserRepository(db)
    user = await repo.get_by_username(username)
    if not user or not getattr(user, "is_fixture_user", False):
        raise HTTPException(404, detail="not a fixture user")
    if not user.is_active:
        raise HTTPException(403, detail="inactive")

    access = create_access_token(user.username, user.role, user.get_permissions())
    refresh = create_refresh_token(user.username)
    await repo.update_refresh_token(user, refresh)
    logger.warning("DEV LOGIN as %s from %s", username, request.client.host)
    return Token(
        access_token=access, refresh_token=refresh, token_type="bearer",
        expires_in=settings.access_token_expire_minutes * 60,
    )
```

In `backend/app/main.py`, after the main router includes:

```python
if settings.enable_dev_auth and settings.environment == "development":
    from app.api import dev_endpoints  # noqa: E402  (intentional conditional import)
    app.include_router(dev_endpoints.router)
    logger.warning("=" * 60)
    logger.warning("DEV AUTH ROUTER MOUNTED — DO NOT RUN IN PRODUCTION")
    logger.warning("=" * 60)
```

Two conditions in both `main.py` and the module itself. If someone removes the `main.py` check but leaves the import, the module-level `assert` still crashes on load. Matches the FastAPI maintainer-recommended conditional-include pattern in [discussion #8088](https://github.com/fastapi/fastapi/discussions/8088).

#### Layer 3 — fixture users with an explicit flag

Alembic migration: add non-nullable `is_fixture_user BOOLEAN DEFAULT FALSE` to `users`. Even if a real user in prod happens to be named "admin", they can't be targeted by `/dev/login-as/admin` because `is_fixture_user` is `False`.

```python
# backend/scripts/seed_dev_users.py
from app.core.config import settings

def seed():
    if settings.environment != "development":
        raise SystemExit("seed_dev_users refuses to run outside development")
    for u in [
        ("dev-admin",   "admin",   "DevAdmin!2026"),
        ("dev-curator", "curator", "DevCurator!2026"),
        ("dev-viewer",  "viewer",  "DevViewer!2026"),
    ]:
        upsert_fixture_user(*u, is_fixture_user=True)
```

This is also the first real runtime caller of `backend/app/auth/user_import_service.py` — a free win that flushes out the dead code.

Fixture passwords live in the dev-only seed script, never in the compiled frontend bundle. The dev endpoint skips password verification for `is_fixture_user=True` accounts: once past Layers 1-2, password checking is theater. Keep the passwords hashed anyway so the same fixture accounts can be used through the regular `/auth/login` endpoint when iterating on the production login UI.

#### Layer 4 — frontend build-time gate

Vite evaluates `import.meta.env.DEV` at **build time** as a literal constant. During `vite build`, esbuild's `define` plugin replaces `import.meta.env.DEV` with `false`; Rollup/esbuild then DCE-eliminates the entire `if (false) { ... }` block, including any imports referenced *only* inside it. Canonical doc: [Vite — Env Variables and Modes](https://vite.dev/guide/env-and-mode).

**Critical gotcha** ([Vite issue #15256](https://github.com/vitejs/vite/issues/15256)): only the built-in `DEV`/`PROD`/`MODE`/`SSR` keys are guaranteed to be statically replaced. **Custom `import.meta.env.VITE_*` flags are NOT guaranteed to tree-shake** — they may survive into the bundle as runtime `process.env` lookups. Always gate on `DEV`, never on a custom `VITE_ENABLE_DEV_LOGIN`.

```vue
<!-- frontend/src/views/Login.vue -->
<script setup>
import { ref, shallowRef, defineAsyncComponent } from 'vue'
// ... existing imports ...

// Async + dynamic import() inside the DEV branch — Rollup drops the chunk in prod
const DevQuickLogin = shallowRef(null)
if (import.meta.env.DEV) {
  DevQuickLogin.value = defineAsyncComponent(
    () => import('@/components/auth/DevQuickLogin.vue')
  )
}
</script>

<template>
  <!-- ... existing login form ... -->
  <component :is="DevQuickLogin" v-if="DevQuickLogin" />
</template>
```

Key tricks:

- **`shallowRef(null)` + dynamic `import()` inside the `if` block** is the only form Vite/Rollup guarantees will produce a separate chunk that gets dropped in prod. A ternary at declaration time works in practice but is more fragile.
- **Never top-level `import DevQuickLogin from ...`** — even if usage is gated, the import keeps the module in the graph because Vue SFCs have side effects (component registration, style injection).
- **Never read `import.meta.env.DEV` into a variable first** and branch on the variable — Vite may not fold through that assignment. Keep the literal `import.meta.env.DEV` inside the `if`.
- **Don't register `DevQuickLogin` globally** in `plugins/vuetify.js` — the global registration call survives DCE. Register it locally inside the dynamic import.

Store action with the same gate:

```js
// frontend/src/stores/authStore.js
async devLoginAs(username) {
  if (!import.meta.env.DEV) return  // tree-shaken in prod — URL string goes with it
  const res = await axios.post(`/api/v2/dev/login-as/${username}`)
  this.setTokens(res.data)
}
```

The `if (!import.meta.env.DEV) return` guard means the URL literal `/api/v2/dev/login-as/` is DCE'd out in prod along with the function body. **No URL string leak in the compiled bundle.**

Verification after every prod build:

```bash
cd frontend && npm run build
# 1. No DevQuickLogin chunk in production dist:
find dist/assets -name '*DevQuick*'      # expect: empty
# 2. No fixture usernames in the bundle:
grep -r "dev-admin\|dev-curator\|dev-viewer" dist/   # expect: empty
# 3. No dev endpoint URL in the bundle:
grep -r "/dev/login-as" dist/            # expect: empty
```

#### Layer 5 — CI + ops guards

Concrete `.github/workflows/ci.yml` steps:

```yaml
- name: Verify dev-auth strings not in prod build
  run: |
    cd frontend && npm run build
    if grep -rq "dev/login-as\|DevQuickLogin\|dev-admin" dist/; then
      echo "::error::Dev-auth strings leaked into production bundle"
      exit 1
    fi
- name: Verify ENABLE_DEV_AUTH not in prod compose
  run: |
    if grep -Eq "^\s*ENABLE_DEV_AUTH\s*[:=]\s*(true|1|yes)" docker-compose.prod.yml; then
      echo "::error::ENABLE_DEV_AUTH must not be truthy in prod compose"
      exit 1
    fi
- name: Staging smoke test — dev endpoint must 404
  run: |
    code=$(curl -s -o /dev/null -w "%{http_code}" https://staging.example/api/v2/dev/login-as/dev-admin)
    test "$code" = "404" || { echo "::error::staging still exposes /dev/login-as"; exit 1; }
```

`docker-compose.prod.yml` explicitly omits `ENABLE_DEV_AUTH` (unset defaults to `False`) **and** sets `ENVIRONMENT=production`, so any runtime override has to fight two defaults.

### 5.4 Bypass-failure matrix

| If this layer fails open … | … this layer still blocks it |
|---|---|
| Dev sets `ENABLE_DEV_AUTH=true` in prod env | **Layer 1** `model_validator` fails startup; process won't boot |
| Layer 1 check removed | **Layer 2** `assert` in module crashes on import |
| Layer 2 check removed | **Layer 3** — no `is_fixture_user=True` rows in prod DB |
| Layer 3 flag defaulted to `True` in a bad migration | **Layer 4** build-time gate strips frontend, endpoint is unreachable from the UI |
| Layer 4 gate accidentally always true | **Layer 1+2** still 404 the endpoint in prod |
| Frontend bundle accidentally ships `DevQuickLogin.vue` | **Layer 2** 404s the endpoint, UI buttons show an error, no login happens |
| Everything fails at once | **Layer 5** CI grep catches it before merge |

No single mistake is sufficient. A successful prod exposure requires all five to fail simultaneously plus the CI check to miss it.

### 5.5 Minimum viable first cut

1. Alembic migration: add `is_fixture_user BOOLEAN DEFAULT FALSE` to `users`.
2. `backend/scripts/seed_dev_users.py` — seeds three fixture users via the existing `UserImportService` (gives that service its first runtime caller).
3. `backend/app/api/dev_endpoints.py` + conditional mount in `main.py`.
4. `backend/app/core/config.py` — `environment` + `enable_dev_auth` + refusal validator.
5. `frontend/src/components/auth/DevQuickLogin.vue` — three Vuetify buttons, stubbed styling.
6. `frontend/src/views/Login.vue` + `authStore.devLoginAs` — dynamic-import gate.
7. `Makefile` target `dev-seed-users`.
8. CI grep checks for the prod compose file + built frontend bundle.
9. One-paragraph note in `README.md`: `ENVIRONMENT=development ENABLE_DEV_AUTH=true make backend`.

**Optional hardening** (not v1):

- Bind backend to `127.0.0.1` only in `make backend` (uvicorn default — instead of `0.0.0.0`).
- Bright-yellow log banner every 60s while the dev router is mounted.
- Fixture passwords read from a gitignored `backend/.env.dev` with sane defaults.
- Playwright test asserting dev buttons present in dev mode and absent in `vite preview` of a prod build.

## 6. Re-prioritized deliverables

Grouped into ship-together bundles. Bundle A is the foundation everything else builds on.

### Bundle A — Foundations (unblock everything downstream)

1. **Dev-mode quick-login** (§5). ~1 day. Faster iteration loop for everything below and finally exercises `user_import_service.py`.
2. **Add CREATE audit entry** in `create_phenopacket()` — small patch to `backend/app/phenopackets/routers/crud.py:312-363`. Prereq for a meaningful revision-history UI.
3. **Add `revision` check to DELETE** (same router). Closes the optimistic-locking gap.
4. **FK-ify audit actor**: Alembic migration converting `changed_by`, `created_by`, `updated_by`, `deleted_by` from `String(100)` → `BigInt FK → users.id` with a one-shot data migration mapping existing usernames. Unblocks every downstream "my reviews / my records" query.
5. **Global soft-delete filter**: SQLAlchemy query event listener or a `PhenopacketQuery` wrapper class every router must go through. Closes the "next router leaks deleted records" risk.

### Bundle B — User management UI (thin wrappers on existing API)

6. **Admin user management view** at `/admin/users` — backend `/api/v2/auth/users*` is already done and tested. Build `frontend/src/views/AdminUsers.vue` with list + filter (role / active / verified) + create + edit + deactivate + role change + resend-invite. Mount it as a card on `AdminDashboard.vue`.
7. **Admin unlock endpoint**: `PATCH /api/v2/auth/users/{id}/unlock` clears `failed_login_attempts` + `locked_until`. Used by the admin UI for account-lockout release.

### Bundle C — Identity lifecycle

8. **Single `credential_tokens` table** (`user_id, purpose ∈ {reset,invite,verify}, token_sha256, expires_at, used_at`). One migration, three flows share it.
9. **Backend endpoints**: `POST /auth/invite` (admin), `POST /auth/invite/accept/{token}`, `POST /auth/password-reset/request`, `POST /auth/password-reset/confirm/{token}`, `POST /auth/verify-email/{token}`. Constant-time anti-enumeration responses per OWASP Forgot Password Cheat Sheet.
10. **Rate-limit binding**: decorate the new endpoints with the existing Redis rate limiter using the norms in §3.3.
11. **Frontend flows**: `ForgotPassword.vue`, `ResetPassword.vue`, `AcceptInvite.vue`. Hook the Login page placeholder alert into the real flow.

### Bundle D — Collaboration layer

12. **`phenopacket.state` enum** + `phenopacket_revisions` immutable snapshot table. 5+1 states per §3.8.
13. **`POST /phenopackets/{id}/transitions`** endpoint with guard matrix per §3.8.
14. **`comments` + `comment_edits` + `comment_mentions`** migrations per §3.8. Simple v1 (no anchor, no parent_id yet — add as nullable later).
15. **`PhenopacketReviewView.vue`** matching the GitHub-PR 3-lane layout from §3.8. Reuse the existing detail view as the left lane.
16. **Diff rendering**: integrate `json-diff-kit-vue` for the Changes tab; server returns the RFC 6902 patch already generated by `backend/app/utils/audit.py`.

### Bundle E — Research identity & attribution (opt-in, privacy-first)

17. **User model ORCID fields** per §3.7.
18. **Authlib ORCID OIDC client** + `POST /auth/orcid/link` / `DELETE /auth/orcid/link` endpoints. `SessionMiddleware` with a strong secret for `state` validation.
19. **`/user/preferences` view** — consent toggles for `orcid_public_opt_in`, display-name preference, `public_attribution_opt_in`. Default all off.
20. **Conditional public attribution** on phenopacket detail pages: render "Reviewed by [name + ORCID]" only when `orcid_source='oauth_authenticated' AND orcid_public_opt_in=TRUE`. Official ORCID icon + full URI per display guidelines.

### Bundle F — Hardening (runs in parallel with C/D/E once Bundle A lands)

21. **`pwdlib` migration** with verify-and-rehash on next login; no forced logouts.
22. **Cookie-based refresh token**: `HttpOnly + Secure + SameSite=Lax` cookie at `/api/v2/auth/refresh`, access token in Vue memory only. Signed double-submit CSRF.
23. **`sessions` table + inventory UI** at `/user/sessions` and the admin variant. Forced logout on role change.
24. **`dependencies=[require_role("admin")]` router-level guard** on `/admin/*`, replacing per-endpoint checks. Closes API5 BFLA.
25. **BOPLA sweep**: replace any `**request.dict()` → `setattr` with explicit Pydantic `UserUpdatePublic` schemas that exclude `role`, `is_superuser`, `refresh_token`.
26. **v-html sanitization audit** on `About.vue` and `FAQ.vue` — route all `renderMarkdown()` output through `utils/sanitize.js`.

### Bundle G — Evidence

27. **Platform security baseline doc** under `docs/security/` — short, written, maps existing controls to OWASP API Top 10 2023 line by line.
28. **Playwright E2E coverage**: login, forgot password, reset, invite accept, admin user CRUD, phenopacket create/edit/delete with audit, review workflow transitions, ORCID link, preferences, session revoke.

### Private contributor dashboard (reserved for a later bundle)

Deliberately not in Bundles A-G because it's a later product decision. Recommended minimum metrics when it lands: records created, records edited, records reviewed, approvals issued, change requests issued, unresolved comments assigned to me, recent activity timeline, weekly/monthly trend. **Private by default**; any future public contributor profile is a separate product decision with fresh privacy review.

## 7. Open questions to answer before Bundle D lands

1. **Public versioning** — should `published → new_draft → published` *replace* the prior public version, or *supplement* it (ClinGen-style)? Determines whether `superseded` is a v1 state or deferred. Recommended default: replace; defer `superseded`.
2. **Unpublished-draft visibility** — curators-only, or logged-in researchers too? Affects whether comments need a per-comment visibility flag. Recommended default: curators-only for v1.
3. **Expert-panel role** — separate ClinGen VCEP-style specialist role beyond admin/curator, or just admin-as-approver? Recommended default: stay with admin/curator; revisit once there are more curators than you can count on one hand.
4. **CRediT role tagging per review event** — low-cost CC-BY taxonomy with high citability upside, but one more field on state transitions. Recommended default: defer to Bundle E.

## 8. tl;dr

- The auth baseline is stronger than the headline suggests — keep refresh-token rotation, the Redis rate limiter, account lockout, security headers, RFC 6902 audit diffs, and the Axios refresh queue. The hardening work is storage + transport + schema, not rewriting rotation.
- Biggest *visible* gap: **user management UI** and **curation history surface**. Both are thin wrappers on code that already exists.
- Ship **Bundle A first** (dev-login + audit-on-create + DELETE revision check + FK audit actor + soft-delete guard). Everything downstream is easier on that base.
- **Bundle B** (admin user UI) is the highest ROI per unit of work.
- **ORCID**: Public API tier + OIDC via Authlib + Phase A (link) before Phase B (sign-in) + opt-in on everything + never email-as-linking-key + never display `admin_curated` or `manual_unverified` iDs publicly.
- **Comments / review / diff**: 5+1 states + one generic `comments` table + RFC 6902 diffs (already generated) + GitHub-PR 3-lane screen. Defer threading, field-pinning, blame, real-time.
- **Dev-mode quick-login** is safe with five layers and is worth building first because it unblocks iteration on everything else.
