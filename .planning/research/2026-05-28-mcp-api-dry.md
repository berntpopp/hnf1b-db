# Keeping the FastMCP server DRY against the FastAPI backend

**Date:** 2026-05-28
**Author:** research (Claude) for Bernt Popp
**Status:** Recommendation — grounded in repo code + FastMCP 3.3.1 / OpenAPI tooling docs
**Scope:** How `mcp/` (FastMCP sidecar) should source the API contract (paths, param
enums, controlled vocabularies, response field names) from the backend OpenAPI
instead of hardcoding it, without violating any LOCKED constraint in
`.planning/specs/2026-05-28-mcp-server-design.md`.

---

## TL;DR recommendation

**Keep the curated, hand-written tool/service/allowlist surface. Do NOT use
`FastMCP.from_openapi` to generate the tools.** Instead, kill drift with a
**two-part anti-drift harness** sourced from `backend`'s `/api/v2/openapi.json`:

1. **Generate** a small `_contract.py` artifact at build time (committed,
   regenerated by `make contract`) holding the path constants and the *machine-readable*
   enums the spec actually exposes (e.g. `comparison`, `sort_by`, `reporting_mode`,
   clinical `sex`/`diabetes_type`, `TransitionRequest.to_state`). Use a tiny
   custom extractor for paths/param-enums plus `datamodel-code-generator` for the
   response-model field names. The allowlist and tool `Literal[...]`s reference
   these constants.
2. **Verify in CI** with a `pytest` contract test that loads `backend`'s live
   `app.openapi()` (no network, no DB) and asserts: every allowlisted path exists
   in the spec; every denied path that exists is intentionally denied; the
   *MCP-defined* vocabularies (the ones the spec can't express as enums — ACMG
   classification, molecular consequence, protein domain) match the backend's
   authoritative Python sets; and the response field names the services read are
   present in the spec's response schemas.

**Why not `from_openapi` for tools:** A decisive empirical finding (below) is that
**the backend does not encode the variant filter enums in OpenAPI** — they are
plain `string` query params validated at runtime against module-level Python
`set`s. `from_openapi` would generate untyped string params and a 1:1 route dump,
discarding exactly the LLM-shaping the spec locked (`response_mode`, `data_class`,
citation, authoritative-content rule, the curated ~11-tool surface). The contract
lives in the backend's *validation modules*, not (yet) in its OpenAPI — so the
single source of truth is "the backend Python contract, surfaced through a
generated artifact + a CI test", not "the OpenAPI tool generator".

Top 3 concrete changes:
- **Add `make contract` + `mcp/src/hnf1b_mcp/contract/_generated.py`** (paths +
  spec-exposed enums + response field-name registry), regenerated from
  `app.openapi()`.
- **Refactor `allowlist.py`, `services/variants.py`, `services/statistics.py`,
  `services/terms.py`, and `tools/*.py`** to import path/enum constants from a
  single `contract` package instead of inlining regexes/`frozenset`/`Literal`.
- **Add `mcp/tests/test_contract.py`** (CI lane) that diffs the MCP's contract
  against the freshly computed `app.openapi()` and against the backend's
  `ALLOWED_*` sets, failing on any add/remove/rename.

## What the repo actually looks like (ground truth)

### Backend OpenAPI

- App: `backend/app/main.py:56` — `FastAPI(openapi_url="/api/v2/openapi.json",
  docs_url="/api/v2/docs", redoc_url="/api/v2/redoc")`. Routers mounted at
  `prefix="/api/v2"` (`main.py:98-110`). So **spec paths are absolute, e.g.
  `/api/v2/phenopackets/aggregate/all-variants`** — note the MCP allowlist/services
  use the `/api/v2`-stripped suffix (the `ApiClient` `base_url` ends in `/api/v2`).
  The contract extractor must strip the `/api/v2` prefix to match the MCP's path
  space.
- The spec is fully obtainable **in-process, offline**:
  `python -c "from app.main import app; app.openapi()"` → 103 paths, 61 component
  schemas. No DB, no network needed. This is the anti-drift oracle.

### What the OpenAPI spec DOES expose as enums (verified by walking `app.openapi()`)

Only **6** `enum` nodes exist, all from `Literal[...]`/`Enum` typed params or schemas:

| Location | enum values |
|---|---|
| `/phenopackets/compare/variant-types` param `comparison` | `truncating_vs_non_truncating`, `truncating_vs_non_truncating_excl_cnv`, `cnv_vs_point_mutation`, `cnv_deletion_vs_duplication` |
| same, `sort_by` | `p_value`, `effect_size`, `prevalence_diff` |
| same, `reporting_mode` | `all_cases`, `reported_only` |
| `/clinical/genital-abnormalities` param | `FEMALE`, `MALE` |
| `/clinical/diabetes` param | `Type 1`, `Type 2`, `MODY` |
| `components.schemas.TransitionRequest.to_state` | `draft`, `in_review`, `changes_requested`, `approved`, `published`, `archived` |

### What the MCP hardcodes that the spec CANNOT give you (the trap)

`backend/app/phenopackets/routers/aggregations/all_variants.py:57-83` declares the
variant filters as `Optional[str] = Query(None, description=...)` — **plain
strings**. Validation is runtime-only, against module-level sets in
`backend/app/phenopackets/variant_search_validation.py`:

- `ALLOWED_CLASSIFICATIONS` = {PATHOGENIC, LIKELY_PATHOGENIC, UNCERTAIN_SIGNIFICANCE, LIKELY_BENIGN, BENIGN}
- `ALLOWED_CONSEQUENCES` = {Frameshift, Nonsense, Missense, Splice Donor, Splice Acceptor, In-frame Deletion, In-frame Insertion, Synonymous, Intronic Variant, Coding Sequence Variant}
- `ALLOWED_VARIANT_TYPES` = {SNV, deletion, duplication, insertion, indel, inversion, CNV}
- `DOMAIN_BOUNDARIES` keys (`all_variants.py:33`) = {Dimerization Domain, POU-Specific Domain, POU Homeodomain, Transactivation Domain}

Consequence: **`PATHOGENIC` appears in the spec only inside a route docstring**
(`description: "...classification: PATHOGENIC, LIKELY_PATHOGENIC, etc."`), never as
a machine-readable `enum`. `datamodel-code-generator` and `from_openapi` are blind
to all four vocabularies.

**Live drift already present (proof the problem is real):** the MCP's `consequence`
enum is `{lof, missense, splicing, inframe, other}` in
`mcp/.../services/variants.py:23` and `tools/variants.py:43-45`, but the backend
accepts `{Frameshift, Nonsense, Missense, ...}`. These do not intersect except by
luck — `consequence="missense"` is silently wrong against the real API. A contract
test would have caught this on day one.

### Other hardcoding surfaces found

- **Paths**: regexes in `client/allowlist.py:7-36`; literal path strings in every
  `services/*.py` (`variants.py:165,204`, `statistics.py:21-32`, `reference.py:40,49,55`,
  `terms.py:97,103`, `publications.py`, `individuals.py:218`, `search.py`).
- **Statistics metric→path map**: `services/statistics.py:21-32` (10 entries) plus
  `_SURVIVAL_COMPARISONS` (`statistics.py:51-53`) duplicating the backend
  `comparison` enum — *this one the spec DOES expose* (see table), so it can be
  generated/verified.
- **Vocabulary names**: `services/terms.py:10-16` lists `sex, interpretation-status,
  progress-status, allelic-state, evidence-code`. These are **path segments** in the
  backend (`app/ontology/routers.py:161-205`: `/vocabularies/sex`, etc.), so the valid
  set IS derivable from spec `paths` keys even though the *values* aren't enums.
- **Response field names**: parsed out of JSON in services — `simple_id, variant_id,
  pathogenicity, molecular_consequence, hg38, structural_type, phenopacket_count`
  (`variants.py:72-85`), `phenopacket_id` / `geneContext.symbol`
  (`individuals.py:34-35,95`), `pp_/var_/pub_/gene_` id prefixes (`search.py:45-55`),
  `GeneDetailSchema`/`domains` (`reference.py`). Many of these (the response models —
  `GeneDetailSchema`, `ProteinDomainsResponse`, the JSON:API item schemas) ARE in
  `components.schemas` and so are generatable/verifiable.

**Net:** the contract splits cleanly into three tiers, which dictates the design:

| Tier | Examples | Source of truth | Strategy |
|---|---|---|---|
| A. In the spec as enums | `comparison`, `sort_by`, `reporting_mode`, clinical sex/diabetes, `to_state` | `app.openapi()` | **generate + verify** |
| B. In the spec structurally | all 103 paths; vocab names as path segments; response field names in `components.schemas` | `app.openapi()` | **generate path consts + verify field names** |
| C. NOT in the spec (runtime-only) | ACMG classification, molecular consequence, protein domain, variant type | backend `ALLOWED_*` Python sets | **verify MCP set == backend set** (import the sets in the test only) |

## Why not `FastMCP.from_openapi` (for tools)

FastMCP 3.3.1 (verified `import fastmcp; fastmcp.__version__ == "3.3.1"`) supports
`FastMCP.from_openapi(openapi_spec, client, route_maps=[...], route_map_fn=...,
mcp_component_fn=..., mcp_names={...})`. It can:

- **Curate a subset**: ordered `RouteMap`s ending in
  `RouteMap(mcp_type=MCPType.EXCLUDE)` ("exclude all others") allowlists specific
  routes. So a curated subset IS possible.
- **Rename/reshape**: `mcp_names` (operationId→name) and `mcp_component_fn(route,
  component)` mutate the generated `OpenAPITool` in place (name/description/tags).
- **Coexist with hand-written tools**: a `from_openapi` server can be combined with a
  normal `FastMCP("…")` (with `@mcp.tool`) via `mount()` (live composition,
  optional `namespace=` prefix). So mixing is supported.

But it is the **wrong tool here**, for reasons specific to this codebase + the LOCKED
spec:

1. **The enums it needs aren't in the spec** (Tier C above). Generated tools would
   take untyped `string` params for classification/consequence/domain — *worse* than
   today's hand-written `Literal[...]`, and the LLM loses its closed enum surface.
2. **It can't express the locked shaping**. `mcp_component_fn` only edits
   name/description/tags/output schema of an already-1:1 tool; it cannot fuse 10
   `/aggregate/*` routes into one `hnf1b_get_statistics(metric=…)` with `dry_run`,
   cannot implement the **authoritative-content rule** (discovery→IDs only, content via
   the head-published path), cannot add `response_mode`/`data_class`/`recommended_citation`,
   and cannot do the cross-tool dedupe/expand the spec mandates. The spec explicitly
   chose "curated, workflow-shaped, NOT a 1:1 dump" — FastMCP's own docs agree
   ("significantly better performance with well-designed and curated MCP servers than
   with auto-converted OpenAPI servers").
3. **It would dissolve the security boundary**. The allowlist in `allowlist.py` is a
   *code-enforced closed-world* gate the spec locked as the security boundary.
   `from_openapi`'s route_maps are a generation-time selection, not a runtime
   enforcement on a shared `httpx` client; the discovery-only flag and the
   authoritative-content invariant have no equivalent.

**Verdict:** stay hand-written for tools/services/allowlist; use OpenAPI only as the
*source of the contract values* (Tier A/B generated, Tier C verified). This is the
"generated *types*, hand-written *tools*" hybrid the task hypothesized — and it's the
right call here precisely because the spec is partly outside OpenAPI.

## Recommended mechanics for this repo

### 1. The oracle: a vendored snapshot + a live recompute

- **CI/test oracle (authoritative):** `from app.main import app; spec = app.openapi()`.
  Runs offline in the `mcp` CI lane only if `backend` is importable there. Since the
  MCP CI lane today has no backend deps, prefer a **committed snapshot** as the
  generation input and the **live recompute** as the verification oracle in the
  *backend* test suite (which already imports the app). Concretely:
  - `backend/tests/test_openapi_contract.py` (new) asserts the committed snapshot
    `mcp/contract/openapi.snapshot.json` equals `app.openapi()` (regenerate on drift).
    This runs in the existing backend `test` job — zero new infra.
  - `mcp/tests/test_contract.py` verifies the MCP's generated constants/enums against
    that committed snapshot (pure file read, runs in the `mcp` lane).
- **Snapshot file:** `mcp/contract/openapi.snapshot.json`, produced by
  `cd backend && uv run python -c "from app.main import app, json; ..."` (a 3-line
  `scripts/dump_openapi.py`). Deterministic (sort keys).

### 2. Generated artifacts (live under `mcp/src/hnf1b_mcp/contract/`)

```
mcp/
  contract/openapi.snapshot.json        # committed input (the contract snapshot)
  src/hnf1b_mcp/contract/
    __init__.py                         # re-exports PATHS, PARAM_ENUMS, RESPONSE_FIELDS
    _generated_paths.py                 # PATHS: dict[str,str] of named path templates
    _generated_enums.py                 # spec-exposed enums (Tier A) as tuples
    _generated_models.py                # datamodel-code-generator pydantic models (Tier B field names)
    vocab.py                            # hand-curated Tier-C sets w/ a "verified against backend" docstring
```

- **Tier A + B (paths, spec enums, vocab path-segments):** a ~60-line custom
  extractor `scripts/gen_contract.py` (in `mcp/`) that loads the snapshot and writes
  `_generated_paths.py` (named constants, e.g. `ALL_VARIANTS =
  "/phenopackets/aggregate/all-variants"`) and `_generated_enums.py`
  (`COMPARISON = ("truncating_vs_non_truncating", ...)`, `SURVIVAL_COMPARISON = ...`,
  `STATISTICS_PATHS = {...}` derived from `/phenopackets/aggregate/*` path keys).
  A custom extractor (not datamodel-codegen) is correct here because we want *path
  constants and param enums*, which datamodel-codegen does not emit (it emits body/
  response *models*).
- **Tier B response models:** `datamodel-code-generator` over the snapshot for the
  response-model field names (`GeneDetailSchema`, `ProteinDomainsResponse`, JSON:API
  item schemas). This gives importable pydantic models so services reference
  `GeneDetailSchema.model_fields` rather than string literals — and the CI test fails
  if a field is renamed/removed upstream. Command (pinned):

  ```bash
  uvx --from 'datamodel-code-generator==0.40.0' datamodel-codegen \
    --input mcp/contract/openapi.snapshot.json --input-file-type openapi \
    --output-model-type pydantic_v2.BaseModel \
    --target-python-version 3.11 --use-standard-collections \
    --enum-field-as-literal all \
    --output mcp/src/hnf1b_mcp/contract/_generated_models.py
  ```
  `--enum-field-as-literal all` emits `typing.Literal` (matches the existing tool
  style); `--output-model-type pydantic_v2.BaseModel` is required (omitting it is
  deprecated since 0.53.0 — pin a current version and pass it explicitly). Output is
  deterministic given a fixed input + pinned version; commit the result and have CI
  re-run + `git diff --exit-code` to catch drift.

- **Tier C (the spec can't express it):** keep the values in `contract/vocab.py` as the
  MCP's declared sets, but make the **backend the source of truth via the test**, not
  by importing backend code into the MCP runtime (LOCKED: no backend import / no DB
  coupling). The backend-side test (`backend/tests/test_openapi_contract.py`) imports
  `ALLOWED_CLASSIFICATIONS`, `ALLOWED_CONSEQUENCES`, `ALLOWED_VARIANT_TYPES`,
  `DOMAIN_BOUNDARIES` and asserts they equal the values committed in the snapshot's
  side-car `mcp/contract/vocab.expected.json`; the MCP test asserts `contract/vocab.py`
  equals that same JSON. Net: one JSON file bridges the two packages with zero import
  coupling, and either side drifting fails CI.

  Even better (optional, higher-value): **fix the backend to type these params as
  `Enum`/`Literal`** so they land in OpenAPI (Tier C→A). That is the truly DRY end
  state — one definition, surfaced in the spec, consumed by generation. Flagged as a
  follow-up because it touches `all_variants.py` + `variant_search_validation.py` and
  is out of the current MCP branch's scope.

### 3. How the allowlist references generated constants

`allowlist.py` keeps its **runtime regex enforcement** (security boundary — unchanged
behavior) but the *patterns are built from* `contract/_generated_paths.py`, and a unit
test asserts every generated path matches exactly one allow rule (or is on the
deny/excluded list). So adding a backend route surfaces as: snapshot test fails →
regenerate → the new path appears unallowlisted → contract test forces an explicit
allow/deny decision. No silent drift, security review still required for each new path.

### 4. Make/CI wiring

Add to `mcp/Makefile`:

```make
contract:                       ## regenerate the contract from backend OpenAPI
	cd ../backend && uv run python ../mcp/scripts/dump_openapi.py > ../mcp/contract/openapi.snapshot.json
	uv run python scripts/gen_contract.py
	uvx --from 'datamodel-code-generator==0.40.0' datamodel-codegen \
	  --input contract/openapi.snapshot.json --input-file-type openapi \
	  --output-model-type pydantic_v2.BaseModel --target-python-version 3.11 \
	  --use-standard-collections --enum-field-as-literal all \
	  --output src/hnf1b_mcp/contract/_generated_models.py
	uv run ruff format src/hnf1b_mcp/contract/
check: lint typecheck test contract-verify
contract-verify:                ## fail if generated artifacts are stale
	$(MAKE) contract && git diff --exit-code -- contract/ src/hnf1b_mcp/contract/
```

CI: the existing `mcp` lane (plan Task 4c) runs `contract-verify` (regenerate +
`git diff --exit-code`) and `pytest tests/test_contract.py`. The existing `backend`
job runs `test_openapi_contract.py` (snapshot == live `app.openapi()`, and `ALLOWED_*`
== `vocab.expected.json`). **Generate-and-commit + verify-in-CI** is chosen over
generate-at-build because the artifacts must be importable, type-checked by mypy, and
reviewable in PRs; the `git diff --exit-code` gate gives the determinism guarantee.

## Migration plan for the already-built code (minimal churn, ordered)

1. **Scaffold the contract package (no behavior change).** Add `scripts/dump_openapi.py`,
   `scripts/gen_contract.py`, commit `contract/openapi.snapshot.json`,
   `contract/vocab.expected.json`, and generate `contract/_generated_paths.py`,
   `_generated_enums.py`, `_generated_models.py`, plus `contract/vocab.py`. Add the
   `make contract`/`contract-verify` targets. — *purely additive.*
2. **Add the CI tests first (red→green safety net):** `mcp/tests/test_contract.py`
   (constants vs snapshot; `vocab.py` vs `vocab.expected.json`) and
   `backend/tests/test_openapi_contract.py` (snapshot vs live `app.openapi()`; backend
   `ALLOWED_*` vs `vocab.expected.json`). This **immediately surfaces the live
   `consequence` drift bug** — fix `vocab.py`/the MCP enum as the first real change.
3. **Refactor `services/statistics.py`** to import `STATISTICS_PATHS`/`SURVIVAL_COMPARISON`
   from `contract` (Tier A — fully spec-backed). Lowest risk, highest payoff (10 paths
   + survival enum de-duplicated).
4. **Refactor `services/terms.py`** vocab list → derive from `contract` path-segments.
5. **Refactor `services/variants.py` + `tools/variants.py`** to import the
   classification/consequence/domain/variant_type sets from `contract/vocab.py` (Tier C)
   and the response field names from `_generated_models.py`. Fix the `consequence`
   values to match the backend.
6. **Refactor remaining services** (`reference.py`, `individuals.py`, `publications.py`,
   `search.py`) to reference `_generated_models` field names + `_generated_paths`
   constants; replace inline path literals.
7. **Refactor `allowlist.py`** last: build the allow regexes from `_generated_paths`,
   keep the deny list + discovery-only flags hand-curated (security), add the
   "every generated path is explicitly allowed-or-denied" test.

Each step is independently testable and leaves the runtime behavior identical except
the `consequence` fix (step 2/5), which is a correctness improvement.

## How the contract test catches drift (risks & limits)

| Upstream change | What fails |
|---|---|
| API **adds** a path | snapshot test (backend) fails → regenerate → new path unallowlisted → `allowlist` test forces explicit allow/deny |
| API **removes/renames** a path | `_generated_paths` const disappears → service import error / contract test fails |
| API **renames a response field** (`pathogenicity`→`classification`) | `_generated_models` field gone → mypy / contract field-name test fails |
| API **changes a spec enum** (`comparison` values) | `_generated_enums` diff → `contract-verify` `git diff` fails |
| API **changes a runtime-only vocab** (ACMG list) | backend test: `ALLOWED_*` ≠ `vocab.expected.json` fails |
| MCP **drifts from backend** (the `consequence` bug) | `vocab.py` ≠ `vocab.expected.json` fails |

**Limitations:**
- Tier C only verifies *set equality against a committed snapshot of the backend's
  Python sets*; it does not prove the backend actually enforces them (it does today via
  `validate_*`). The real fix is typing those params as enums (Tier C→A follow-up).
- The snapshot introduces a "regenerate" step on every legitimate API change; mitigated
  by `make contract` being one command and CI telling you exactly what drifted.
- `datamodel-code-generator` output can shift across versions → **pin the version** and
  gate on `git diff --exit-code`.
- FastMCP `from_openapi` is intentionally *not* on the runtime path, so we don't inherit
  its 1:1-dump or untyped-param limitations; we only borrow its philosophy (curate, don't
  dump).
