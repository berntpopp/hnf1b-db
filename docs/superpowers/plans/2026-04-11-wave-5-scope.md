## 1. Executive summary

Wave 4 finished the backend decomposition work ‚Äî every file under `backend/app/` is now below 500 LOC, the phenopacket CRUD layer has a proper Router ‚Üí Service ‚Üí Repository stack, admin sync task state is cache-backed, and 907 backend tests pass with 8 HTTP surface baseline fixtures locking the API response shape. Wave 5 builds on that foundation to **prepare HNF1B-DB to graduate from an invited-curator data-entry tool to a credible multi-user curation platform**.

The canonical input for Wave 5 scope is the 2026-04-11 Platform Readiness Review (`docs/reviews/2026-04-11-platform-readiness-review.md`, untracked). That review proposes 7 bundles (A‚ÄìG) totalling ~28 deliverables. Wave 5 deliberately takes a subset:

- **Bundle A ‚Äî Foundations** (audit-on-create, DELETE revision check, FK-ify audit actor, global soft-delete filter, dev-mode quick-login)
- **Bundle B ‚Äî User Management UI** (admin user CRUD + unlock)
- **Bundle C ‚Äî Identity Lifecycle** (credential tokens + invite / reset / verify endpoints + frontend flows, invite-only)
- **Top of Bundle F ‚Äî Hardening** (BOPLA sweep, BFLA router guard, pwdlib migration with verify-and-rehash, v-html sanitization audit)
- **Three targeted 2026-04-09 frontend decomposition items** that directly block Bundle B: split `frontend/src/api/index.js` (953 LOC), extract `useSyncTask` composable from `AdminDashboard.vue` (905 LOC), fix v-html XSS in `About.vue` + `FAQ.vue`

**Explicitly deferred to Wave 6** (¬ß6 below): Bundle D (comments / review state machine / 3-lane review screen), Bundle E (ORCID + preferences + public attribution), rest of Bundle F (cookie-based refresh token, sessions table, session inventory UI), Bundle G (security baseline doc, Playwright E2E expansion), the four remaining 2026-04-09 giant frontend components (`PageVariant.vue`, `HNF1BGeneVisualization.vue`, `ProteinStructure3D.vue`, `useSeoMeta.js`), MFA/TOTP/passkeys, private contributor dashboard, SMTP infrastructure, CRediT role tagging.

Wave 5 is done when an admin can create curator accounts from the UI, a user who forgot their password can reset it without an admin intervening, every phenopacket CRUD operation leaves a FK'd audit row, the v-html XSS from the 2026-04-09 review is closed, all existing bcrypt-hashed users can log in without forced logouts, and `grep dev-admin dist/` returns empty on a production build.

## 2. Goal and shape

### 2.1 Goal

> Prepare HNF1B-DB to graduate from *"invited-curator data-entry tool"* to *"multi-user curation platform with credible identity lifecycle"* ‚Äî without shipping comments, review states, ORCID, or full session hardening (those are Wave 6).

### 2.2 Shape

- **3 sequential PRs** to main.
- **3 sibling git worktrees** per the `CLAUDE.md` convention:
  - `~/development/hnf1b-db.worktrees/chore-wave-5a-foundations/`
  - `~/development/hnf1b-db.worktrees/chore-wave-5b-user-management/`
  - `~/development/hnf1b-db.worktrees/chore-wave-5c-identity-lifecycle/`
- Worktrees B and C are created *after* the predecessor has merged to main, rebased on fresh main.
- **Reviewability ceiling:** each PR ‚â§ 15 commits / ‚â§ 50 files / ‚â§ +40 tests. Wave 4 was 27/~97/+28 ‚Äî single-PR ceiling. Wave 5 stays below that per PR.
- **HTTP surface baselines** renamed from `backend/tests/fixtures/wave4_http_baselines/` ‚Üí `backend/tests/fixtures/http_baselines/` in PR 1 commit 1 (refactor-only), extended with new fixtures in each subsequent PR.
- **Tech-debt register** at `docs/refactor/tech-debt.md` continues; expect exactly one new entry in PR 2 for the pwdlib bcrypt-verifier-fallback transition period.
- **Wave 5 exit notes:** `docs/refactor/wave-5a-exit.md`, `docs/refactor/wave-5b-exit.md`, `docs/refactor/wave-5c-exit.md` per PR, plus a consolidated `docs/refactor/wave-5-exit.md` at the end of PR 3.

### 2.3 Process rules

- Sequential branches, not stacked. PR 2 is not created until PR 1 has merged to main. PR 3 is not created until PR 2 has merged.
- Alembic migrations: every migration has a reversible `downgrade()` written and tested in CI before the upgrade is applied to the dev DB.
- Commit discipline: atomic commits per item, no "misc fixes" commits, no drive-by refactors outside the scope table below.
- Informal merge freeze during Wave 5 execution ‚Äî non-critical unrelated work is held until Wave 5 lands, to avoid rebase churn on the schema migrations.

## 5. Risks and mitigations

### R1 ‚Äî A4 audit-actor FK migration (HIGH)

Converting 4 `String(100)` columns to `BigInt FK ‚Üí users.id` across an existing dataset where some rows hold strings that don't map to any user ‚Äî notably the `"hnf1b-db v1.0.0"` string from the data migration, confirmed visible in the Playwright walk (review ¬ß4 item 8).

**Mitigation 1:** Reserved placeholder user created in the same migration ‚Äî `username='_system_migration_'`, `is_active=False`, `role='viewer'`, seeded via `op.bulk_insert()` in the `upgrade()` step.
**Mitigation 2:** Keep all 4 FK columns nullable (matches current `Optional[str]`). Only values that fail to map become NULL ‚Äî never the placeholder, unless the distinct username was the literal `"hnf1b-db v1.0.0"` (in which case it maps to `_system_migration_`).
**Mitigation 3:** Write `downgrade()` before `upgrade()` and test it in CI ‚Äî convert FK back to username string via `SELECT u.username FROM users u WHERE u.id = phenopackets.created_by_id`.
**Mitigation 4:** Before merging PR 1: manual `pg_dump` snapshot of the local hybrid DB, run the migration, walk the UI, confirm all audit rows still display correctly, run `alembic downgrade -1`, confirm state is exactly restored.
**Mitigation 5:** `PhenopacketRepository` query updates that JOIN on the FK get an explicit test that validates the resulting JSON payload matches the baseline fixtures.

### R2 ‚Äî pwdlib migration regression (HIGH)

Risk: users with existing `$2b$12$...` bcrypt-hashed passwords can't log in after the passlib ‚Üí pwdlib swap.

**Mitigation 1:** `pwdlib.PasswordHash.recommended()` returns `PasswordHash([Argon2Hasher, BcryptHasher])` ‚Äî the bcrypt fallback verifier handles all bcrypt rounds 4‚Äì17. Explicitly tested in `test_pwdlib_rehash.py` with a pre-computed `$2b$12$...` hash pasted into the test.
**Mitigation 2:** Existing `backend/tests/test_auth.py` must continue passing without modification. Any test that depends on `passlib.hash.bcrypt` directly gets updated to use `pwdlib.hashers.bcrypt.BcryptHasher`.
**Mitigation 3:** Staging smoke test before prod merge: create a test user with a bcrypt password via the *pre-merge* build, upgrade to pwdlib, confirm same user can log in, stored hash is now Argon2id.
**Mitigation 4:** Tech-debt register entry documents that the bcrypt verifier fallback is permanent runtime cost until all live users have logged in at least once post-migration.

### R3 ‚Äî Dev-mode quick-login leaking to prod (CRITICAL if it happens)

Any of the 5 layers failing silently is disaster. The layered defense *is* the mitigation, but we also need positive evidence each layer works.

**Layer 1 test:** `test_config_refuses_dev_auth_in_prod.py` ‚Äî `Settings(environment="production", enable_dev_auth=True)` raises `ValidationError`.
**Layer 2 test:** `test_dev_endpoints_loopback_only.py` ‚Äî request with non-loopback `client.host` returns 403.
**Layer 3 test:** `test_dev_login_requires_fixture_user.py` ‚Äî non-fixture user returns 404 even with `is_active=True`.
**Layer 4 verification:** CI grep jobs run on the *production* build, not the dev build ‚Äî `ENVIRONMENT=production npm run build` then `grep -r "dev-admin\|DevQuickLogin\|dev/login-as" dist/` ‚Äî exit 1 on any match.
**Layer 5 verification:** Staging smoke test scaffolded so it can activate once staging exists ‚Äî hits `/api/v2/dev/login-as/dev-admin` and asserts 404.
**Additional mitigation:** PR 1 exit criteria include a manual checklist item: "Reviewer verified the CI log for the `vite build` job shows zero dev-auth strings in `dist/`."

### R4 ‚Äî Router-level BFLA guard accidentally allowing previously-guarded routes (HIGH)

Moving from per-endpoint `Depends(require_admin)` to router-level `dependencies=[...]` is behavior-preserving in theory but a regression risk in practice.

**Mitigation 1:** Three-commit sequence. Commit 1 ‚Äî add `test_admin_route_authorization.py` with explicit viewer‚Üí403 / curator‚Üí403 / admin‚Üí200 assertions for every admin route; confirm it passes against per-endpoint guards. Commit 2 ‚Äî apply router-level guard; run the same tests; nothing changes. Commit 3 ‚Äî remove the per-endpoint guards; run tests again; nothing changes.
**Mitigation 2:** The three-commit sequence lets us `git revert` the last commit if anything breaks.

### R5 ‚Äî `api/index.js` split breaking the refresh queue (MEDIUM)

The `isRefreshing` + `failedRequestsQueue` module-level state coordinates concurrent token refreshes. Splitting the axios instance from the queue state is a thunder-herd reintroduction risk.

**Mitigation 1:** `transport.js` owns BOTH the axios instance AND the refresh-queue state. These are co-located. `session.js` owns tokens + storage and has no axios dependency.
**Mitigation 2:** Existing `frontend/tests/unit/stores/authStore.spec.js` (22 tests per review ¬ß2.3) runs against the split modules without modification.
**Mitigation 3:** New `transport.spec.js` test fires 5 concurrent 401 responses and asserts the refresh endpoint is called exactly once.

### R6 ‚Äî Global soft-delete filter over-application (MEDIUM)

An unscoped query event listener that adds `WHERE deleted_at IS NULL` to every query will hide rows from admin endpoints that intentionally want deleted records (audit browser, future "restore" workflow).

**Mitigation 1:** Filter is scoped to the `Phenopacket` entity class only, not every model.
**Mitigation 2:** The Phenopacket detail audit endpoint (`crud.py:545-612`) uses an explicit `include_deleted=True` parameter that bypasses the filter.
**Mitigation 3:** Test parametrized over every phenopacket read-router confirms deleted rows don't leak, PLUS a test confirming the audit endpoint CAN see deleted rows.

### R7 ‚Äî PR 1 commit budget overflow (MEDIUM)

Dev-mode quick-login + FK migration + audit-on-create + DELETE revision + soft-delete filter + v-html is a lot. 14 commits is tight.

**Mitigation 1:** Atomic commit per item. No "misc fixes" commits.
**Mitigation 2:** If commit 15 is needed, the overflow signals splitting PR 1. Most likely split point: dev-mode quick-login moves to its own sub-PR between 5a and 5b. Decision is taken at PR 1 commit 13, not earlier.
**Mitigation 3:** Refactor-only commits (baseline directory rename) go first. Schema changes second. Behavior changes last.

### R8 ‚Äî Non-Wave-5 merges to main during Wave 5 execution (MEDIUM)

Sequential PR strategy means PR 2 and PR 3 each rebase on fresh main. Unrelated work landing on main during Wave 5 compounds rebase pain on the schema migrations.

**Mitigation 1:** Informal merge freeze on non-critical work during Wave 5, communicated in PR 1's description.
**Mitigation 2:** PR 2 worktree is created and rebased immediately after PR 1 merges, before any other work can land.
**Mitigation 3:** HTTP baseline tests act as the safety net ‚Äî any accidental response-shape drift from unrelated work shows up as a fixture diff.

### R9 ‚Äî Email dispatch infrastructure does not exist (MEDIUM, PR 3 only)

PR 3 needs to send invite / reset / verify emails but HNF1B-DB has no email infrastructure.

**Mitigation 1:** `backend/app/auth/email.py` is an abstraction with `ConsoleEmailSender` (stdout, dev default) and `SMTPEmailSender` (aiosmtplib, prod option). Selection via `settings.email_backend ‚àà {"console","smtp"}`.
**Mitigation 2:** PR 3 ships `ConsoleEmailSender` as default for dev + CI. Real SMTP is a post-Wave-5 ops task.
**Mitigation 3:** `test_email_backend.py` verifies both implementations produce the same payload structure (URL, token, expiry, user context).
**Mitigation 4:** Integration tests in `test_auth_invite.py` use `ConsoleEmailSender` and assert the URL/token appears in captured log output (via `pytest caplog`).
